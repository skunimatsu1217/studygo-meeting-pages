変更前後の比較

既存の checkInNote フィールドを3つに分割します

❌ 変更前（現在）
DBフィールド
Shop.checkInNote — 1つのみ
管理画面
「チェックイン方法」テキストエリア 1つ
API レスポンス
checkInNote: string | null
アプリ表示
全ユーザーに同じメッセージ
出し分け
不可能
✅ 変更後（新規）
DBフィールド
Shop.checkInNote — その他向け（既存を流用）
Shop.checkInNoteMonthly — 月額向け（新規）
Shop.checkInNotePeriodPass — 期間パス向け（新規）
管理画面
プランタイプ別テキストエリア 3つ
API レスポンス
checkInNote: string | null（プランに応じた値を返す）
アプリ表示
プランタイプに応じたメッセージを表示
出し分け
月額 / 期間パス / その他 の3パターン
データマイグレーション戦略
既存の checkInNote はそのまま「その他（ドロップイン等）」向けとして残します。
新規フィールド checkInNoteMonthly, checkInNotePeriodPass はNULLデフォルトで追加。
→ 既存データへの影響なし。管理者が新フィールドを設定するまで、月額・期間パスユーザーにも既存メッセージが表示されます。
フォールバック 月額向け/期間パス向けが未入力の場合 → 既存の checkInNote を表示
checkInNote も未入力 → メッセージなし（現在と同じ）
理由 マイグレーション直後、まだ新フィールドが未入力でも既存の動作が壊れない

実装概要

変更が必要なファイル一覧と変更内容

1. DBスキーマ変更
ファイルprisma/schema.prisma
変更内容 Shopモデルに2フィールド追加：
checkInNoteMonthly String? @db.MediumText
checkInNotePeriodPass String? @db.MediumText
マイグレーションnpx prisma migrate dev --name add_plan_checkin_notes
2. API バックエンド
CheckInService.ts チェックイン時にプランタイプを判定し、対応する checkInNote を選択して返す
MONTHLY → checkInNoteMonthly ?? checkInNote
PERIOD_PASS → checkInNotePeriodPass ?? checkInNote
その他 → checkInNote
ShopInfoService.ts チェックイン中の情報取得時にも同じロジックで出し分け
adminShopResolver.ts 新フィールドの取得・更新に対応
CompanyShopResolver.ts 新フィールドの取得・更新に対応
definition.ts APIレスポンス型に変更なし（checkInNoteのままサーバー側で出し分け）
3. フロントエンド（管理画面）
admin/shops/validation.tsx 既存の checkInNote ラベルを「ドロップイン等向け」に変更
checkInNoteMonthly, checkInNotePeriodPass のバリデーション追加
admin/shops/types.ts 型定義に checkInNoteMonthly, checkInNotePeriodPass 追加
alliance/shops/config.tsx Company側にも同じフィールド追加
4. Flutterアプリ
変更なし APIレスポンスの checkInNote フィールドはそのまま。
サーバー側で出し分けた結果がそのまま返るため、アプリ側の変更は不要。
表示ロジック（AppBoxNotification）も現状のまま動作する。
5. 影響範囲まとめ
変更するもの API: schema.prisma, CheckInService, ShopInfoService, adminShopResolver, CompanyShopResolver
Front: validation.tsx, types.ts, config.tsx
変更しないもの Flutter: アプリ側は変更不要（サーバーが出し分けるため）
API定義: definition.ts のレスポンス型は変更なし
既存データ: 既存の checkInNote はそのまま「その他」向けとして機能
破壊的変更 なし（後方互換性あり）
