スタディGO — 実装仕様書（外注用）｜会議決定事項反映済み｜通知以外ほぼ確定

期間パスとは？

決まった日数のあいだ、追加料金なしで使い放題になる「買い切り」のパスです。
月額プランと違い、自動更新はありません。期間が終われば自動的に終了します。
お客様が自分で好きなタイミングに、好きな日数のパスを購入できます。

決定項目の進捗 0 / 0

決定済み: 店舗が自由に日数を設定可能

店舗（加盟法人ではなく店舗単位）が自由に日数を設定してパスを作成できる方式に決定。
最短0日（当日）〜最長365日。種類数は制限なし。
ただし1人が同じ店舗で同時に購入できるパスは1つまで。入会金は不要。
追加要望: 各パスに販売可能枚数を設定でき、上限到達時はグレーアウト表示にしたい。

  • 自由に設定する場合、最短は何日から？
  • 自由に設定する場合、最長は何日まで？
  • 1つの店舗で同時に何種類のパスを販売できるか？

外注開発時に必要なテーブル設計です。既存のPrismaスキーマに追加します。

period_pass_plans（パスプラン定義）
idInt, PK, auto increment

durationDaysInt — 日数（0〜365）
priceInt — 税込価格（円）、最低50円
maxSalesInt? — 販売枠上限（null=無制限）
currentSalesInt, default 0 — 現在の販売数

sortOrderInt — 表示順

createdAt / updatedAtDateTime

idInt, PK, auto increment
userIdInt, FK → users.id
planIdInt, FK → period_pass_plans.id
shopIdInt, FK → shops.id
startDateDate — 利用開始日
endDateDate — 利用終了日（startDate + durationDays - 1）
priceInt — 購入時の価格（税込）
couponIdInt? — 使用クーポン
discountAmountInt, default 0 — 割引額
statusEnum: ACTIVE / EXPIRED / TERMINATED（運営強制終了）
stripePaymentIdString — Stripe決済ID
checkinCountInt, default 0 — 利用回数
terminatedAtDateTime? — 強制終了日時
terminatedByInt? — 強制終了した管理者ID
createdAtDateTime — 購入日時

主要APIエンドポイント

GET /api/shops/:id/period-passes店舗の期間パスプラン一覧取得（ユーザー向け、isActive=trueのみ）
POST /api/period-passes/purchaseパス購入（Stripe決済→契約作成）。body: { planId, startDate, couponCode? }
GET /api/users/me/period-passesログインユーザーの契約一覧（ACTIVE/EXPIRED含む）
POST /api/checkin既存チェックインAPIを拡張。期間パス→月額→ドロップインの優先順位で判定
POST /api/admin/period-pass-plans運営: プラン作成。body: { shopId, name, durationDays, price, maxSales? }
PUT /api/admin/period-pass-plans/:id運営: プラン更新（契約者がいる場合は新規作成を促す）
PUT /api/admin/contracts/:id/terminate運営: 契約の強制終了
GET /api/admin/period-pass/analytics運営: 販売数・売上・利用率の集計

決済失敗時の処理

カード決済が失敗した場合は既存の決済エラーフローに合わせる。
エラー表示 → 「再試行」or「カード変更」ボタン。契約レコードは作成しない。

手数料計算ロジック

デフォルト手数料率月額プランと同じ料率を適用
個別設定法人ごとに期間パス専用の手数料率を設定可能
計算式店舗売上 = 購入価格 × (1 - 手数料率)
例: ¥3,500 × (1 - 0.20) = ¥2,800 が店舗売上
精算サイクル月末締め → 翌月末払い（既存の精算フローに統合）
売上計上日購入日（決済完了日）

契約のルール

お客様がパスを購入してから期限が切れるまでのルールを決めます。
キャンセル・延長・再購入などの取り扱いも含みます。

決定済み: 契約ルール（14/14項目すべて確定）

✅ 開始日: カレンダー表示、今日〜30日後まで選択可、デフォルトは今日
✅ 定休日表示: 月額プランの仕様と同じ
✅ 途中解約: 不可（返金なし）
✅ 臨時休業時: パス期間は延長しない
✅ 再購入: 期限切れ後のみ。延長ではなく別途購入（違う日数も可）
✅ 連続購入: 回数制限なし。再購入時もクーポン利用可
✅ チェックイン優先順位: 期間パス → 月額 → ドロップイン
✅ 期限切れ後チェックイン: プラン選択画面を表示（ドロップイン→期間パス→月額の順）

実装仕様: 契約ライフサイクル（状態遷移） ▼

期間パス契約は以下の状態を持ちます。日次バッチで有効期限チェックを行います。

●
ACTIVE（有効）
購入完了〜endDateの23:59:59まで。チェックイン時に追加料金なし。
マイページに残日数・進捗バー表示。
▼ endDate 23:59:59 経過（日次バッチ）
●
EXPIRED（期限切れ）
自動遷移。マイページに「利用可能日数: 0日」表示。
7日間表示後、非表示に。再購入を促すバナー表示。
▼ 運営が手動操作した場合
●
TERMINATED（強制終了）
運営のみ実行可能。terminatedAt, terminatedByを記録。
返金は別途対応（Stripe管理画面から手動返金）。

日次バッチ処理（cron）

実行タイミング毎日 00:05 JST
処理内容endDate < 今日 かつ status=ACTIVE の契約を EXPIRED に更新
通知連動残り3日/1日/当日の契約に対して通知をキューに追加（通知パネルで詳細決定）

管理画面の設計

店舗スタッフや運営側が使う管理画面でのパスの設定・確認方法を決めます。

決定済み: 管理画面仕様（5決定/3却下）

✅ プラン作成・料金設定: 運営（本部）のみ
❌ 法人による有効/無効切替: 今は不可（却下）
✅ 料金変更: 契約者なし→直接変更 / 利用履歴あり→新プラン作り直し

❌ 売上レポート画面: 却下（既存レポートに統合）
❌ 期間パスvs月額比較グラフ: 却下

重要: 運営による強制契約・解約機能

手動期限切れや手動延長は不要だが、運営が特定ユーザーに対してパスを強制的に付与（契約）または解約できる機能が必要。
用途: トラブル対応、返金対応後の契約終了、テスト用の契約作成など。

導入計画

どこから始めるか・いつまでに何をするかを決めます。

決定済み: 導入計画（5/5項目すべて確定）

✅ 導入方式: 一部店舗でテスト → 段階的に拡大（オプション同様に管理画面から作成）
✅ テスト対象: 勉強カフェ 新宿スタジオ、勉強カフェ 新宿BPスタジオ
✅ 優先度: 他の機能と並行もしくは先にこちら
✅ 利用規約作成: 根本さん担当
✅ アカウント削除時: 警告表示 → 削除可能 → 返金なし
