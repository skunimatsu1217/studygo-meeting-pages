1. 背景・目的

1.1 現状の課題

課題項目現状問題点
ユーザー登録ステップ数6ステップ・13項目の入力が必要離脱率が高い。特にカード入力で離脱する傾向
認証方式メール+パスワードのみSNSログインに慣れたユーザーの獲得が困難
決済方法クレジットカードのみ（Stripe）カード非保有者・PayPayユーザーの取り込み不可
見学予約の導線アプリインストール必須Webからの流入（広告・SNS経由）を取り込めない
見学予約フロー4画面遷移（カレンダー→利用期間→アンケート→確認）利用期間・アンケートが見学段階では不要で煩雑

1.2 改修の目的

🎯 目的1：登録障壁の低減

SNSログインにより1タップで認証を完了させ、段階的に情報収集することで、初回の入力項目を最小限（氏名+電話番号）に削減する。

🎯 目的2：決済手段の多様化

PayPay決済を追加し、クレジットカードを持たないユーザーやQRコード決済を好むユーザーにも対応する。

🎯 目的3：Web経由の集客強化

アプリ不要のWeb見学予約ページを提供し、広告・SNSからの直接予約を可能にする。

🎯 目的4：見学予約のCVR向上

予約フローを4画面から2画面に短縮し、不要なアンケート項目を任意化することで完了率を改善する。

🎯 目的5：既存ユーザーへの影響ゼロ

既存のメール+パスワード認証・Stripe決済は完全に維持し、あくまで選択肢の追加として実装する。

3.4 サーバー側処理

// auth/google.controller.ts

@Post('/auth/google')
async googleLogin(@Body() body: { idToken: string }) {
  // 1. Google IDトークンを検証
  const ticket = await oAuth2Client.verifyIdToken({
    idToken: body.idToken,
    audience: GOOGLE_CLIENT_ID,
  });
  const payload = ticket.getPayload();
  // payload.sub = Google固有ID, payload.email, payload.name

  // 2. 既存ユーザー検索（googleId → email → 新規作成）
  let user = await AppUserRepository.findByGoogleId(payload.sub);
  if (!user) {
    user = await AppUserRepository.findByEmail(payload.email);
    if (user) {
      // 既存メールユーザーにgoogleIdを紐付け
      await AppUserRepository.update(user.id, { googleId: payload.sub });
    } else {
      // 3. 新規作成（最低限の情報のみ）
      user = await AppUserRepository.create({
        email: payload.email,
        googleId: payload.sub,
        firstName: payload.given_name,
        lastName: payload.family_name,
        authProvider: 'google',
        password: randomHash(), // ダミー（SNSユーザーは使用しない）
        status: 'registered',
      });
    }
  }

  // 4. JWTトークン発行
  return { token: generateJwt(user), isNewUser: !user.isRegistered };
}

4.4 サーバー側処理

// auth/line.controller.ts

@Post('/auth/line')
async lineLogin(@Body() body: { code: string }) {
  // 1. 認可コード → アクセストークン交換
  const tokenRes = await axios.post(
    'https://api.line.me/oauth2/v2.1/token',
    {
      grant_type: 'authorization_code',
      code: body.code,
      redirect_uri: LINE_CALLBACK_URL,
      client_id: LINE_CHANNEL_ID,
      client_secret: LINE_CHANNEL_SECRET,
    }
  );

  // 2. IDトークンからメール取得（取得可能な場合）
  const idToken = tokenRes.data.id_token;
  const decoded = jwt.decode(idToken);
  const email = decoded?.email; // 未設定 or 権限なしの場合はundefined

  // 3. プロフィール取得
  const profileRes = await axios.get(
    'https://api.line.me/v2/profile',
    { headers: { Authorization: `Bearer ${tokenRes.data.access_token}` } }
  );

  // 4. ユーザー検索 or 作成
  let user = await AppUserRepository.findByLineId(profileRes.data.userId);
  if (!user) {
    user = await AppUserRepository.create({
      lineId: profileRes.data.userId,
      email: email || `line_${profileRes.data.userId}@placeholder.studygo.jp`,
      firstName: profileRes.data.displayName,
      authProvider: 'line',
      password: randomHash(),
      status: 'registered',
    });
  }

  return { token: generateJwt(user), isNewUser: !user.isRegistered };
}

4.5 LINE Login メール取得の制限

注意: LINEではメールアドレスの取得に制限がある
  • LINEにメールアドレスを登録していないユーザーからはメールを取得できない
  • メール取得には LINE Developers コンソールで「メールアドレス取得権限」の申請が必要
  • メール未取得時はプレースホルダーメール（line_{userId}@placeholder.studygo.jp）を設定し、後からStage 2で実メールを収集する設計

// payment/paypay.service.ts

async createPayPayPayment(userId: number, amount: number, orderId: string) {
  const payload = {
    merchantPaymentId: orderId,
    amount: { amount, currency: 'JPY' },
    codeType: 'ORDER_QR',
    orderDescription: 'StudyGO 入会料金',
    redirectUrl: 'https://studygo.jp/payment/complete',
    redirectType: 'WEB_LINK',
    userAgent: 'StudyGO App',
  };

  const response = await paypayClient.qrCodeCreate(payload);

  return {
    paymentUrl: response.BODY.url, // PayPayアプリへのリダイレクトURL
    merchantPaymentId: orderId,
    expiry: response.BODY.expiryDate,
  };
}

// Webhook受信
async handleWebhook(payload: PayPayWebhook) {
  if (payload.notification_type === 'Transaction') {
    const payment = await getPaymentDetails(payload.merchantPaymentId);
    if (payment.status === 'COMPLETED') {
      // 契約作成・ステータス更新
      await ShopContractService.completeContract(payload.merchantPaymentId);
    }
  }
}

5.5 定期課金の制限事項

重要: PayPayは自動定期課金に対応していない
  • PayPay Web Payment APIでは 都度決済 のみ対応。StripeのようなSubscription（自動定期課金）は不可
  • 月次課金は 毎月PayPayアプリで手動承認 が必要になる（UXが悪い）
  • 対応策: PayPayは 初回決済（入会金+初月+翌月） でのみ使用し、3ヶ月目以降はクレジットカード登録を促す設計とする
  • 代替案: PayPay継続課金APIの利用申請を行う（審査あり、承認率は不明）

5.6 PayPay × Stripe の併用方針

14. リスクと注意事項

14.1 アカウント競合

リスク: 同一ユーザーが複数の認証方法でアカウントを作成する可能性
  • ケース1: メールで登録済みのユーザーが、同じメールのGoogleアカウントでログイン → メールアドレスで自動紐付け
  • ケース2: Googleで登録後、LINEでもログイン（メールが異なる）→ 別アカウントとして作成される
  • 対策: マイページに「アカウント連携」機能を設け、後からGoogle/LINE/メールを統合できるようにする（Phase 4で実装）

14.2 PayPay定期課金の制限

注意: PayPayは月次自動課金に非対応
  • 初回決済はPayPayで可能だが、3ヶ月目以降のサブスクリプションはクレジットカードが必要
  • PayPay利用者には、契約2ヶ月目にカード登録を促すプッシュ通知・アプリ内バナーを表示
  • カード未登録で3ヶ月目を迎えた場合は、アプリ利用時にカード登録モーダルを強制表示

14.3 スパム対策（WEB見学予約）

対策詳細
reCAPTCHA v3フォーム送信時にスコア判定。0.5未満は予約を拒否
レート制限同一IP/メールアドレスから1時間に3件まで
メール確認予約確認メールのリンククリックで予約を確定（オプション）
管理画面通知大量予約を検知した場合に管理者へアラート

14.4 セキュリティ考慮事項

項目対策
IDトークン偽造Google: verifyIdTokenで署名検証。LINE: API経由でトークン交換するため改ざん不可
CSRFOAuth state パラメータを使用。ランダム文字列を生成しセッションに保存
PayPay Webhook偽装PayPayが提供するHMAC署名ヘッダーを検証
WEB予約のBot攻撃reCAPTCHA v3 + レート制限 + メール認証の3層防御
個人情報保護WebPreviewReservationのメール・電話番号は暗号化保存を検討

14.5 その他の注意事項

項目内容
App Store / Google Play 審査SNSログイン追加時は審査でリジェクトされる可能性あり。Apple Sign-Inの追加が必要になる場合がある（Appleポリシー）
LINE Developersの審査メールアドレス取得権限の申請に1〜2週間かかる場合がある
